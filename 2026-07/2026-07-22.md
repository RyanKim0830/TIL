## 1. 클라이언트 · 서버 · API
- **API는 "요청을 받는 쪽"에 존재한다.** (서버가 창구를 열어두는 것)
- **Client** = 요청을 *보내는* 쪽 / **Server** = 요청을 *받는* 쪽
- **API를 "부르는" 로직 = Client.** (남의 API를 호출하는 코드는 클라이언트 쪽)
- 작업 흐름: **API 명세서(스펙)를 먼저 정하고 → 그 기능을 구현**한다. (spec-first)

## 2. 네트워크 방향 & 데이터 흐름
- **인바운드(inbound)** = 데이터가 외부 → 내부로 *들어옴*
- **아웃바운드(outbound)** = 데이터가 내부 → 외부로 *나감*

### Push vs Pull
| | 누가 움직이나 | 언제 |
|---|---|---|
| **Push** | 데이터를 *가진* 쪽이 능동적으로 보냄 | 변화가 생긴 순간 |
| **Pull** | 데이터가 *필요한* 쪽이 가지러 감 | 주기적으로 확인 |

- **판단 기준: "변화가 언제 일어나고, 누가 그 순간을 아는가."**
  → 가진 쪽이 변화를 안다면 Push / 필요한 쪽이 확인해야 안다면 Pull


## 3. HTTP 요청 형식
### JSON vs Multipart
| | JSON | Multipart |
|---|---|---|
| 용도 | 텍스트 데이터 | 파일 + 텍스트 함께 |
| Content-Type | `application/json` | `multipart/form-data` |
| 파일 업로드 | 부적합 | 표준 방식 |

- **Multipart** = 하나의 HTTP 요청 안에 *여러 종류의 데이터*를 함께 담는 형식
- **쿼리 파라미터**: `params = {}` (dict)로 넘기면 httpx가 알아서 `?a=1&b=2`로 붙여줌
- **URL 인코딩**: 파일명 등을 URL *경로*에 넣을 땐 인코딩 필요 (한글·공백·특수문자 깨짐 방지) → `quote()`
